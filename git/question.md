## 本地创建仓库推至云端
### 00由来
&emsp;&emsp;之前, 学过一端时间的`git`, 不过最后化繁为简, 只是记住了推送到云端的命令行, 而且一直在使用. 不过当时创建仓库还需要在网页上创建, 然后拉回本地的. 那个时候整体上也十分混乱, 没考虑其它的东西. <br>
&emsp;&emsp;后来, 觉得本地可以直接创建仓库, 然后推送至云端. 学习之后呢, 又固化为了几个命令行, 并且存储在某个地方. 需要的时候`copy`即可. <br>
&emsp;&emsp;因为是`copy`, 不用动脑子, 原理也没分析过. 但每次需要新创建仓库时, 总会再去查找写好的命令行, 并且按照操作来就行.<br>
&emsp;&emsp;根据我自己的学习哲学, 这样单纯的重复, 并没有什么用, 因为没有留下心理表征, 每次需要时, 总会再去看. 一开始我的观点是, 用的少肯定就记不住, 用的多了就自然而然的记住了. 但现实似乎并不是这样. 因为用的多, 不代表会留下心理表征. 记住的前提条件, 我觉得还是留下印象极为深刻的心理表征. 这也是我创建这个仓库的目的.<br>
### 00原理
- 整体思想非常简单, 就是先在本地创建仓库, 然后去云端创建仓库, 接着建立联系即可. 我们先来看看本地创建仓库. 
~~~git
git init
git add .
add commit -m "初次提交"
~~~
- `git init`, 就是在本地生成一个`.git` 隐藏文件夹, 包含各种与`Git` 相关的元数据, 比如版本历史, 配置信息等. 这样之后, 我们就可以添加文件, 提交更改, 管理分支, 等等. 本地创建仓库, 这句肯定是必须的. 
- `git add .`, 代表将当前目录提交到暂存区. 但是需要注意, `git add .` 跟`ait add -A`, 还是存在一定区别的, 不过目前我还没有搞清楚, 知道即可.
- `git commit -m` 则是将缓存区的信息, 提交到本地仓库中.
    - 分析. 不论是在本地, 还是在云端, 有些概念是一定要懂的. 别人设计好的东西, 肯定有自己的原理所在, 一些思想没搞懂的话, 单纯看命令行, 就是很难理解的. 所以, 这里我们分析一下必要的理解词汇.
    - 工作区. 这个就很简单, 其实就是我们编写代码的地方.
    - 本地仓库. 这个其实是`git` 中的一个概念. 云端仓库, 就很明确, 在`github` 网址中的. 那么本地仓库, 其实就是我们本地的一个仓库. 建立联系, 并不是将我们的工作区跟云端建立联系, 而是将本地仓库跟云端仓库建立联系. 所以, 我们在工作区编写好了代码后, 需要先提交到本地仓库, 然后再提交到云端仓库.
    - 暂存区. 这个就是工作区与本地仓库的一个缓存区域. `git add` 系列操作, 其实就是将工作区中的文件(其实也不是文件, 可能是相关指针什么的, 这里就用文件表述吧), 提交到缓存区. 而`git commit` 系列操作, 则是将缓存区的文件再提交到本地仓库.
    - 有了这些基础呢, 我们再回看上面说的, 其实就很明确了. 而且, 每个命令行都是有其存在的意义的. 
        - 一开始的`git init`, 是在工作区里面创建了`.git` 文件, 但是这样不行, 我们需要提交到本地仓库中.
        - 想要提交到本地仓库, 需要先提交到暂存区, 也就是`git add` 操作.
        - 将暂存区的内容提交到本地参考, 则需要`git commit`操作.
- 这个时候, 我们的本地仓库已经创建好了. 但是还远远不够, 我们需要在云端建立仓库, 建立好了之后, 获取其网址或者`ssh`, 然后执行如下命令行.
~~~git
git remote add origin https...
~~~
- 这个命令, 其实很深奥. `git remote` 是一个子命令, 用于本地仓库跟远程仓库建立联系的. `origin`, 并不是命令中自带的. 这是给远程仓库命的名. 我们也可以起其它的名字. 但是需要注意, 之后我们使用的`git push origin main`, 这个`origin`, 就是指的远程仓库名. 如果改了, 这里也需要改. `origin` 是约定俗成的. 后面就是仓库的`url` 了.
- 我们可以使用`git remote -v`, 查看远程仓库名, 也可以通过`git remote show origin`, 查看更具体信息.
- 此时, 这里的`git push origin` 等, 才有了其存在的意义, 也就是将本地仓库推送至名为`origin` 的远程仓库. 根据`url`可以获取远程地址.
~~~git
git branch -M main
~~~
- 这个命令行, 就是将当前分支名字改为`main`. 我们习惯这样. 新建仓库的话, 分支名会是`master`? (我忘了, 但确实不是`main`). 其实这句是可有可无的. 但建议还是这么操作.
~~~git
git push -u origin main
~~~
- 这句命令行, 我们其实很熟悉了. 对于`origin main`, 其实就是远程仓库的`main`分支. 如果远程没有`main` 分支, 就会自动创建. 它很跟当前我们执行这个命令行的本地分支建立联系(一般是`main`对`main`). 
- 我们可以通过多个命令行查看谁与谁建立的联系. `git status`, `git branch -vv`, `git remote show origin`.
- 这样, 我们就完整的将本地仓库与云端仓库建立了联系. 并且, 通过这次的深入分析, 明白了分支之间如何建立的联系. 当然, 关于分支, 不再这里了的讨论范围内, 以后用到的时候再分析.
- 我们再总结下

|  命令行  | 含义 |
|----------|------|
|git init|本地创建`.git`| 
|git add .|提交至缓存区| 
|git commit -m "初次提交"|提交至本地仓库|
|git remote add origin https...|与远程建立联系|
|git branch -M main|将本地分支改名为main|
|git push -u origin main|将远程main跟本地main建立联系|
